Roles & Permissions

HomeFinder uses four user roles. Roles are stored in the users.role column of each site's database. The master role is the super-admin and exists only once — only palarson holds this role.

Capability User Agent Owner Master
Browse listings, maps, digest
AI deal analysis & portfolio
Save preferences permanently
Manage clients & set client prefs
Masquerade as a client
Override AI prompts (per-agent)
Approve / suspend agents
View API cost metrics
Override AI prompts (global)
Trigger manual pipeline fetch
Create & manage sites (multi-tenant)
Never demote your master account. The migration script runs on every app start and will restore palarson to at least owner — but only the database holds the master role. If you need to inspect or correct roles, use SQLite directly: UPDATE users SET role='master' WHERE username='palarson';
The is_admin check in code is true for both agent, owner, and master roles. The is_owner check is true for owner and master. is_master is true only for master.
Site Management

Each HomeFinder "site" is an independent market with its own SQLite database, listing data, users, and scores. Sites are tracked in instance/registry.db. Site management requires the master role.

Creating a new site
  • 1Navigate to Sites in the top nav (master only). Click Map Picker to set the center location and select zip codes visually. Type a city name in the search box and press Enter to jump the map.
  • 2Click zip code polygons on the map to include them (green). Click again to remove. The bounding box auto-computes from your selection.
  • 3Click Apply to Form. This writes the lat/lon, zoom, zip codes, and SW/NE bounds into the create form.
  • 4Fill in Site Key (lowercase, e.g. atlanta), Display Name, and Owner Email. Click Create Site & Initialize Database.
  • 5The system creates the SQLite database, initializes all tables, and seeds your master account as owner in the new site DB.
  • 6Run the pipeline once to populate listings: click Fetch Now from the dashboard of the new site.
Site keys are permanent. The key becomes part of the cookie value and URL routing. Changing it after launch will log out all existing users of that site. Choose a short, obvious slug: charleston, bathwv, atlanta.
Site configuration fields
site_keyUnique slug. Used in the hf_site cookie and X-HomeFinder-Site header. Lowercase letters, numbers, underscores only.
display_nameHuman-readable name shown in the navbar and page titles. e.g. "Bath, West Virginia".
db_pathRelative path from project root to the SQLite database. Stored as e.g. instance/bathwv.db. Do not change after creation unless you also move the file.
zip_codes_jsonJSON array of zip code strings. Used to seed the Preferences page target areas and constrain the pipeline's search area.
map_bounds_json[[sw_lat, sw_lon], [ne_lat, ne_lon]] bounding box. Constrains the census place autocomplete and the map default viewport.
map_center_lat/lonDefault map center for the listing map page and preferences map.
owner_emailInformational — the email of the owner user registered in this site's DB. Not used for access control.
Activating and deactivating sites

Use the pause/play button on each site card. Inactive sites are skipped by the nightly scheduler pipeline. Users with a valid hf_site cookie for an inactive site will be routed to the default site as a fallback.

How URL routing works

There are two ways to route a request to a specific site's database:

  1. URL path prefix — /site/<key>/ in the path. The SitePathMiddleware strips this and sets the X-HomeFinder-Site environ header before Flask sees the request.
  2. Cookie — hf_site=<key>. Set by the /go/<site_key> route when a user clicks a market on the landing page. No site key appears in any URL. Lasts 30 days.
The landing page at /welcome uses the cookie method, so users always see clean URLs like /auth/login instead of /site/charleston/auth/login.
Agent Management

Agents register via the Apply as an Operator link on the login page. They start in pending status and cannot log in until approved.

Approving an agent
  • 1Go to Admin → Agents. Pending agents appear at the top with a warning badge.
  • 2Review the agent's name, license number, brokerage, and phone number.
  • 3Click Approve. The agent can now log in and access their dashboard.
  • 4Optionally expand the agent's row to add owner notes: contract signed, background checked, MLS verified checkboxes, plus a free-text notes field.
Suspending an agent

Click Suspend on any approved agent. Suspended agents receive a "your account has been suspended" message at login. Their existing clients are not affected — clients retain their accounts and data but lose the agent assignment functionality.

Agent branding

Agents can customize the navbar color, add a logo URL or emoji icon, and set a tagline. Clients of that agent see the branded navbar. As owner you cannot set an agent's branding — only the agent can do this from their dashboard.

Client accounts are created by the agent (not by the client registering themselves). The agent uses My Clients → Add New Client — HomeFinder generates a username and temporary password and emails the client automatically.
Running the Pipeline

The pipeline fetches listings from Zillow and Realtor.com, geocodes them, assigns Census place names, scores them, and marks removed listings as inactive. It is the only mechanism that updates listing data.

Manual trigger (from the UI)

Click Fetch Now from the dashboard home page. This button is visible to owners and agents. The pipeline runs synchronously — the page will not respond until it completes (typically 30–120 seconds depending on listing volume).

Manual trigger (command line)
cd C:\path\to\home_finder_agents
activate.bat
python run.py pipeline --site charleston

Omit --site to run for all active sites.

Pipeline steps in order
  • 1Fetch — Calls Zillow API and Realtor.com API for the site's zip codes. Returns raw listing objects.
  • 2Deduplicate — Merges Zillow and Realtor.com results by address. Prefers Realtor.com for detail fields.
  • 3Geocode — Adds lat/lon for any listing missing coordinates, using the Google Geocoding API.
  • 4Upsert — Inserts new listings or updates existing ones. Preserves first_seen. Updates price_history_json on price changes.
  • 5Place name — Assigns a Census TIGER place name (neighborhood/city) to each listing based on coordinates.
  • 6Score — Runs the 17-factor scorer for every listing without an up-to-date score.
  • 7Mark inactive — Any listing that did not appear in this run's results is marked status=inactive.
  • 8Update registry — Writes pipeline_last_run and listing_count back to the registry.
Lazy detail enrichment: Full property details (room details, HOA, photos) are NOT fetched during the pipeline. They are fetched on-demand the first time a user opens a listing's detail page (details_fetched=False triggers the fetch and caches the result).
Nightly Scheduler

APScheduler runs the pipeline automatically at 3:00 AM server time for all active sites. It runs as a background thread inside the Waitress process.

Confirming the scheduler is running

At app startup you will see this in the log:

Scheduler started — nightly pipeline at 3:00 AM

If instead you see:

APScheduler not installed — nightly pipeline disabled.

Run: pip install apscheduler --break-system-packages inside the virtualenv, then restart the app.

IIS / Waitress restarts

When IIS recycles the app pool or Waitress restarts, APScheduler re-registers the job on startup. No manual intervention is needed. However, if the app is down at 3 AM, that night's pipeline run is skipped — there is no catch-up mechanism. You can manually trigger Fetch Now the next morning.

The scheduler uses a daemon thread. It does not prevent the process from exiting normally. On IIS this is fine — Waitress handles the shutdown gracefully via the atexit handler registered at startup.
AI Prompt Overrides

All three AI features (Deal Brief, Portfolio Analysis, Preferences Coach) have default system prompts defined in config.py. Owners can override these globally; agents can further override them for their own clients.

Priority order
  1. Agent override — if an agent has a custom prompt for this type, it is used for all AI calls made by that agent's clients.
  2. Global owner override — if no agent override exists but you have set a global override via Admin → Prompts, it is used.
  3. Default (config.py) — used when no override exists at either level.
The three prompt types
dealSystem prompt for single-listing Deal Brief analysis. Must instruct the model to return JSON with keys: summary, strengths, concerns, negotiation, verdict.
portfolioSystem prompt for multi-listing Portfolio Analysis. Must return JSON with keys: headline, ranking, patterns, strategy, dark_horse, bottom_line.
preferencesSystem prompt for Preferences Coach analysis. Must return JSON with keys: headline, strengths, blind_spots, tweaks, local_insight, bottom_line.
Always keep the JSON instruction in your overrides. If you remove the instruction to respond with a JSON object, the AI will return plain text and the UI will display an "unexpected format" error. The content and persona of the prompt can be changed freely — just keep the JSON structure requirement.
Clearing an override

Leave the text field blank and click Save. The override row is deleted from the database and the default from config.py is used on the next AI call.

Environment Variables

All secrets are loaded from .env in the project root via python-dotenv. Never commit .env to source control.

SECRET_KEYFlask session signing key. Generate with python -c "import secrets; print(secrets.token_hex(32))". Changing this logs out all users.
ANTHROPIC_API_KEYRequired for all AI features. Obtain from console.anthropic.com. If missing, AI buttons show an error message but the app continues to function.
RAPIDAPI_KEYUsed for Zillow and Realtor.com API calls via RapidAPI. Required for the pipeline to return listings.
GOOGLE_MAPS_API_KEYUsed for geocoding (coordinates from addresses) and Google Places detail enrichment. If missing, geocoding is skipped and detail pages may lack some fields.
MAIL_SERVER / MAIL_PORTSMTP configuration for transactional email (verification, password reset, welcome). MAIL_USE_TLS, MAIL_USERNAME, MAIL_PASSWORD, MAIL_DEFAULT_SENDER also required.
HOMEFINDER_REGISTRYOptional. Absolute path to registry.db. Defaults to instance/registry.db relative to the project root. Only set this if you need the registry in a non-standard location.
Troubleshooting
Login takes me to the wrong site

Your browser has an hf_site cookie pointing to a different site. Clear it in DevTools → Application → Cookies, or visit /home_finder_agents/go/<correct_site_key> to set the right cookie.

Pipeline runs but returns 0 listings
  • Check that RAPIDAPI_KEY is set correctly in .env.
  • Verify the site's zip codes are valid and within the supported region (SC fully supported; WV partially supported via state-level fallback).
  • Confirm your RapidAPI subscription is active for both Zillow and Realtor endpoints.
AI analysis returns "unexpected format" error

The AI returned plain text instead of JSON. This usually means a prompt override is missing the JSON instruction. Go to Admin → Prompts, check each override, and ensure it instructs the model to respond "ONLY with a JSON object." Clearing the override to restore the default also fixes this.

My master role keeps changing to owner

This was a known bug in the migration script — now fixed. The old code used role != 'owner' which demoted master on every startup. The fix uses role NOT IN ('owner', 'master'). Deploy the updated __init__.py.

SQLite WAL lock errors in logs

The registry uses WAL mode to reduce locking. If you see database is locked errors during pipeline runs, it usually means a previous pipeline is still running. The site DB connections use check_same_thread=False which is safe for Waitress's multi-threaded model.

After IIS deploy, old pages still appear

Python caches .pyc bytecode. After replacing Python files on the server, touch web.config (open and save with no changes) to force IIS to recycle the app pool, or delete all __pycache__ directories and restart Waitress.

All startup activity, site routing decisions, and scheduler events are logged at WARNING level so they appear in the IIS log even with the default log level. Check the IIS application log at C:\inetpub\logs\LogFiles\W3SVC1\ for a full trace of what happened on the last request.
